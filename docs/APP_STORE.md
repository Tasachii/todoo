# TodoDesu — App Store Guide

> **Status: shelved.** This path was fully prepared and works, but distribution
> currently goes through the free GitHub Pages deployment + PWA instead (no Apple
> Developer fee). Everything below stays valid for the day a native build is wanted.

How to build the native iOS app and ship it to the App Store. Everything in the
repository is already prepared; the only things that need your Mac are Xcode and an
Apple Developer account.

## How the native app works

The iOS app is the same React app wrapped with **Capacitor**, with one key difference:
there is no Fastify server on a phone, so the app runs in **standalone mode** — a local
data engine (`packages/web/src/api/local.js`) that implements the exact same business
rules as the server (sort order, soft delete + 30-day purge, single focus session,
duration capping) against a JSON snapshot in on-device storage.

The switch is automatic and per-build (`packages/web/src/api/client.js`):

| Environment | Data lives in |
|---|---|
| Browser / PWA (normal `npm run build`) | SQLite on your Mac, via the REST API |
| Native app (Capacitor) or `VITE_STANDALONE=1` build | On-device storage inside the app |

The two stores do **not** sync in v1 — the phone app is its own island, by design
(sync is on the roadmap, after cloud deployment + auth). The standalone engine is fully
covered by tests (`packages/web/test/local.test.js`).

## Prerequisites (one-time)

1. **Xcode** from the Mac App Store (the full app, not just Command Line Tools), then:
   ```bash
   sudo xcode-select -s /Applications/Xcode.app/Contents/Developer
   ```
2. **Apple Developer Program** membership ($99/year) — [developer.apple.com/programs](https://developer.apple.com/programs/)
3. No CocoaPods needed — the project uses Swift Package Manager.

## Building and running

```bash
npm run app:sync   # standalone web build → copy into the iOS project → restore normal build
npm run app:open   # open the project in Xcode
```

In Xcode (first time only):

1. Select the **App** target → **Signing & Capabilities** → choose your **Team**.
2. The bundle id is `com.rawinlab.todoo` — change it if you prefer (it must be unique
   on the App Store and match the App Store Connect record you create below).
3. Pick a simulator or your plugged-in iPhone and press **Run** to try it.

Re-run `npm run app:sync` whenever the web app changes, then build again in Xcode.

## Shipping to the App Store

### 1. Create the app record
On [App Store Connect](https://appstoreconnect.apple.com) → My Apps → **+** → New App:
iOS, name **Todoo** (or your variant — names are globally unique), the bundle id from
above, SKU anything (e.g. `todoo-001`).

### 2. Archive and upload
In Xcode: select **Any iOS Device (arm64)** as the destination → **Product → Archive**
→ in the Organizer, **Distribute App → App Store Connect → Upload**. Signing is
automatic. The export-compliance question is pre-answered: `Info.plist` declares
`ITSAppUsesNonExemptEncryption = NO` (the app uses no non-exempt encryption — there is
no networking at all in standalone mode).

### 3. TestFlight first (recommended)
The uploaded build appears under TestFlight within ~15 minutes. Add yourself as an
internal tester and live with the app for a few days before submitting.

### 4. Fill in the listing
- **Category:** Productivity
- **Screenshots:** required for 6.9" (iPhone Pro Max) and 13" (iPad Pro); take them in
  the Simulator with **⌘S**. Show Today, Board, Focus (the ring), and dark mode.
- **Description:** lead with what it is — a quiet, minimal todo app with a board,
  calendar, and Pomodoro focus timer; everything stays on your device.
- **Privacy:** App Privacy → **Data Not Collected** (true: no accounts, no analytics,
  no network calls; all data stays on the device).
- **Age rating:** the questionnaire yields 4+.

### 5. Submit for review
Attach the build to version 1.0, add review notes, submit. Reviews typically take
1–2 days.

**Review notes worth writing** (guideline 4.2, minimum functionality — reviewers are
wary of thin web wrappers): state that the app is fully functional offline, all features
(tasks, kanban board, calendar, Pomodoro focus with statistics) run entirely on-device
with no server dependency, and no account is needed.

## Releasing updates

1. Make changes, run `npm run app:sync`.
2. In Xcode bump **Version** (e.g. 1.0 → 1.1) on the App target — keep
   `package.json` versions in step for sanity.
3. Archive → upload → attach the new build to a new version in App Store Connect.

## Icons

The App Store icon (1024×1024, opaque RGB — Apple rejects icons with an alpha channel)
is generated by the same zero-dependency script as the PWA icons and already placed in
the asset catalog:

```bash
node scripts/gen-icons.mjs   # regenerate after changing the design
cp packages/web/public/icons/icon-1024.png \
   "packages/web/ios/App/App/Assets.xcassets/AppIcon.appiconset/AppIcon-512@2x.png"
```

## Practical notes

- **Where the data lives:** WKWebView localStorage inside the app's container. Deleting
  the app deletes the data. If the app ever grows beyond a personal todo list, migrate
  the engine's storage adapter to a native plugin (it takes an injected storage object
  precisely for this). In the unlikely event localStorage is unavailable the engine
  falls back to in-memory storage — the app still works but data lasts only for that
  launch; a corrupt snapshot is preserved under `todoo-data-v1.corrupt` before the
  engine starts fresh.
- **`app:sync` always restores `dist/`** to the normal (server-backed) web build as its
  last step, even if the sync itself fails — so `npm start` after a failed sync still
  serves the right bundle.
- **Fonts:** loaded from Google Fonts; with no network the app falls back to system
  fonts and remains fully usable. Self-host the fonts if pixel-perfect offline
  typography ever matters.
- **What is committed:** the Xcode project (`packages/web/ios/`) is in git; generated
  things (`App/public/`, user state, DerivedData) are gitignored. `cap sync` regenerates
  them.
- **Android later:** the same standalone mode means `npx cap add android` is all the
  app-side work; the rest is Play Console process.
